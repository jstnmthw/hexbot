# HexBot IRC Logic & Reasoning Audit — 2026-04-11

Audit against IRC realities (RFC 1459/2812/7194, IRCv3, network diversity:
Libera/Solanum, OFTC/Hybrid, UnrealIRCd, InspIRCd, ngIRCd, IRCnet, EFnet).
Findings reference `file:line`. Severity: **P0** (correctness/security bug) /
**P1** (compat risk) / **P2** (polish).

Scope: protocol handling, state tracking, mode ops, output framing, CAP/SASL,
permissions, lifecycle. Botlink and DB internals are only spot-checked.

---

## 1. Message splitting & output framing

`src/utils/split-message.ts:5,14-54`

- **P0** — `splitMessage` uses JavaScript `.length` (UTF-16 code units), not
  bytes. IRC's 512-byte limit counts bytes. A line of 400 emoji is ~1600 bytes
  and will be truncated mid-codepoint by the server. Any channel using
  Cyrillic/Japanese/emoji will hit this.
- **P0** — `substring(0, splitIdx)` can cut **mid-UTF-16 surrogate pair** and
  mid-grapheme-cluster. Surrogate halves become mojibake on the wire.
- **P1** — 400 "char" budget is fixed; doesn't subtract the actual outbound
  prefix length (`:nick!user@host PRIVMSG #chan :`). On networks with long
  cloaks (Libera `user/foo` or vhosts) real room is smaller.
- **P1** — Not used for CTCP/ACTION payload framing; CTCP needs `\x01...\x01`
  wrappers counted in the byte budget.
- **P2** — `MAX_LINES=4` hard truncation is a policy call, but appending
  `" ..."` to the last line can itself push past the byte limit.

### Todo

- [ ] Replace `.length` with `Buffer.byteLength(str, 'utf8')`.
- [ ] Iterate by grapheme cluster (or at least codepoint via `Array.from`) to
      never split a surrogate pair.
- [ ] Accept a `reservedBytes` param computed from current
      `nick!user@host` + target + verb + `" :"` + `\r\n`.
- [ ] Document CTCP/ACTION framing math; splitter should be told the line is
      CTCP.
- [ ] Add a test with Cyrillic + emoji + ZWJ sequences (skin-tone, family).

## 2. Mode batching

`src/core/irc-commands.ts:139-155`

- **P0** — `mode(channel, '+o-v', 'a', 'b')` breaks: `direction = charAt(0)`
  captures only `+`, then `slice(1).split('')` yields `['o','-','v']`,
  producing a single batch that re-applies `+` to all three chars. Mixed
  directions must be segmented.
- **P0** — The batcher assumes **every mode char takes one param**. Callers
  can produce wrong lines (e.g. `mode('#c','+oo','a')` sends `+oo a`, second
  `o` silently dropped). There is no length validation.
- **P1** — `modesPerLine` defaults to 4 but is never fed from ISUPPORT
  `MODES=` on 005; InspIRCd commonly advertises 20, Unreal 12, Solanum 4.
- **P1** — No `CHANMODES` parsing. The bot cannot know per network which
  modes take params (A=list, B=setting, C=setting when +, D=flag).

### Todo

- [ ] Forbid mixed-direction input in `mode()`; force callers to batch `+`
      and `-` separately, or walk tokens and emit one segment per direction
      switch.
- [ ] Parse ISUPPORT `CHANMODES` and `PREFIX` at 005; expose a helper
      `expectsParam(modeChar, direction)` so batching knows param counts.
- [ ] Pull `MODES=` from ISUPPORT into `setModesPerLine()` automatically.
- [ ] Integration tests against two `CHANMODES` dialects (Solanum vs
      InspIRCd).

## 3. Channel-state mode parsing

`src/core/channel-state.ts:265-336`

- **P2** — `onNick` only rekeys users in shared channels (line 254, correct),

## 5. IRCv3 capabilities & SASL

`src/bot.ts:510-540`, `src/core/services.ts`

- **P0** — `sasl_mechanism` defaults to `PLAIN`. On networks without TLS,
  PLAIN **leaks the password**. `tls_verify: true` is defaulted correctly
  (line 511), but nothing asserts `tls === true` when SASL PLAIN is
  configured.
- **P1** — Only `PLAIN` and `EXTERNAL` supported. Missing: `SCRAM-SHA-256`
  (Atheme), `ECDSA-NIST256P-CHALLENGE` (Libera). Not blockers;
  PLAIN-over-TLS is fine in practice.
- **P1** — No explicit cap negotiation list; entirely delegated to
  irc-framework. Hexbot only flips `enable_chghost`. Worth **consciously**
  requesting:
  - `multi-prefix` — **biggest quick win**. Without it, NAMES only returns
    the highest prefix, so a user with `+o+v` is tracked as `o` only and
    voice state is wrong for ops-with-voice.
  - `userhost-in-names` — avoids WHO spam on join.
  - `away-notify` — zero-cost channel-state improvement.
  - `account-tag` — account on every PRIVMSG without WHO.
  - `server-time` — accurate timestamps across reconnects for relay/log
    plugins.
  - `message-tags` — prerequisite for modern features.
  - `echo-message` — dedup on labeled-response paths.
  - `batch`, `labeled-response` — synchronous-ish request/response.
  - `setname`, `draft/chathistory` — bouncer ergonomics.
  - `STS` — see §9. Ignoring STS on a network that advertises it is a
    security regression on reconnect.
- **P1** — NickServ IDENTIFY fallback after SASL: verify ordering. If SASL
  succeeds but `services.ts` also sends IDENTIFY, we burn one NickServ slot
  and potentially leak creds.
- **P2** — No re-request of caps after `CAP NEW` / `CAP DEL`.

### Todo

- [ ] Hard error at startup if `sasl_mechanism === 'PLAIN'` and
      `tls !== true`.
- [ ] Explicitly request: `multi-prefix`, `userhost-in-names`, `away-notify`,
      `account-tag`, `server-time`, `message-tags`, `echo-message`, `batch`,
      `labeled-response`. Confirm each is consumed or gracefully ignored.
- [ ] Implement STS: persist directive and bounce to TLS on reconnect; refuse
      to downgrade if a prior STS was seen within its duration.
- [ ] Audit SASL → NickServ fallback path; confirm no double-auth.
- [ ] Handle `CAP NEW` / `CAP DEL` after registration.
- [ ] Optional: add SCRAM-SHA-256 as a third mechanism.

## 7. Permissions & identity

`src/core/permissions.ts:180-270,370-386`

- **P0** — `checkFlags` only matches by **hostmask**. Modern networks use
  vhosts/cloaks; hostmasks are less reliable than account names. There is no
  way to say "user `foo` has +o in `#dev` _by account_". Given that
  `extended-join`/`account-notify`/`account-tag` are already captured into
  `networkAccounts`, the permission layer should accept `$a:accountname` as
  a pattern form.
- **P0** — `findByNick` (line 210) matches only the **nick portion** of the
  pattern against the raw nick. Any stored `foo!*@*` lets an attacker just
  take the nick `foo` and match. CLAUDE.md says "warn on insecure masks"
  and the code warns at add time (370-386), but `findByNick` itself performs
  a lookup that is _less safe than the warning implies_.
- **P1** — Dispatcher's NickServ ACC verification uses a cached account map
  fast path (good). Verify cache invalidation on `NICK`, `QUIT`,
  `account-notify -`, and `PART` from the last mutual channel — `onNick`
  rekeys (244-252), but `onQuit`/`onPart` paths need verification so stale
  auth does not survive re-connect.
- **P1** — No "bot-only" flag vs "owner" distinction. `botlink-*` reuses the
  same flag namespace — confirm leaf-authorized commands cannot be escalated
  via the IRC side.
- **P2** — Per-channel overrides exist; missing is a `.chanset`-style
  convenience to grant a per-channel flag without editing the full user.

### Todo

- [ ] Accept `$a:accountname` (or similar) in `record.hostmasks`; on
      `checkFlags`, look up `ctx.account` from `networkAccounts` and match
      these alongside hostmask patterns.
- [ ] Deprecate or delete `findByNick`; callers must use a full hostmask.
- [ ] Verify `networkAccounts` invalidation across `onQuit`, `onPart`,
      reconnect.
- [ ] Audit botlink command relay path: confirm leaf bots cannot trigger
      permission changes on hub without the hub re-validating through its
      own flags.
- [ ] Test: attacker takes a nick, joins from a different host, tries a
      `+n`-gated command → rejected.

## 9. Connection lifecycle

`src/core/connection-lifecycle.ts:84-150,257-274`

- **P1** — Startup backoff `1000 * 2^attempt` capped at 30s + jitter is fine,
  but **does not honor server reconnect hints**. On a K-line/G-line, servers
  send `ERROR :Closing Link ... (K-Lined)`. The bot retries in 30s forever.
  Detect `K-Lined`/`G-Lined`/`Banned` in the ERROR reason and back off to
  ≥1h or stop.
- **P1** — No `STS` honoring (see §5).
- **P1** — `auto_reconnect_max_retries: 10` total — confirm what happens at
  exhaustion. Process exit is correct for supervisord/docker restart; make
  sure the exit is intentional and non-zero.
- **P2** — "Periodic presence check every 30 min" (257-274) is a reasonable
  kludge, but `JOIN` storms can follow netsplits. Trigger only after a
  plausible split.
- **P2** — No differentiation of `ERROR :Excess Flood`. If hexbot is
  flooding, the sane response is to halve `messagesPerSecond` for the next
  session, not reconnect at the same rate.

### Todo

- [ ] Parse `ERROR` reason and apply policy: K/G-line → long backoff or
      stop; Excess Flood → halve send rate next session; throttle → short
      pause.
- [ ] Implement STS (persist duration + port; enforce TLS within window).
- [ ] Document exit-on-exhaustion behavior for ops.
- [ ] Trigger rejoin sweep only after a plausible split, not every 30 min.

## What Eggdrop does (relevant prior art)

- **Account-based flags**: Eggdrop stores userfile records keyed by handle;
  hostmasks are just aliases. Match by account (via `CHANACC` / NickServ)
  is preferred when available. HexBot is currently handle+hostmask only and
  should add account patterns.
- **Mode batching**: Eggdrop queues mode changes per channel and flushes as
  one `MODE` line per batch, tracking per-mode param requirements from a
  hardcoded network table. A modern port should instead parse `CHANMODES`.
- **Flood protection**: Eggdrop has separate `flood-msg`, `flood-chan`,
  `flood-ctcp`, `flood-join`, `flood-deop`, `flood-kick`, `flood-nick`
  counters per-host. HexBot's unified flood system is a reasonable
  simplification; evaluate per-type for CTCP and joins specifically.
- **STS / SASL / IRCv3**: Eggdrop only recently shipped these. Follow the
  IRCv3 spec directly. `multi-prefix` is the single biggest quick win.

## Appendix A — Audit adjustments from web verification (2026-04-11)

After writing the audit, I re-verified the load-bearing claims against primary
sources (RFCs, IRCv3 specs, irc-framework source). Most held; two findings
need correction and one new **P0** was uncovered.

### A.1 Confirmed against primary sources

- **§1 (512-byte limit)** — RFC 1459 §2.3: "IRC messages are always lines of
  characters terminated with a CR-LF pair, and these messages shall not
  exceed 512 characters in length, counting all characters including the
  trailing CR-LF. Thus, there are 510 characters maximum allowed for the
  command and its parameters." The claim that hexbot's `splitMessage`
  undercounts bytes via JS `.length` stands unchanged.
- **§2 (CHANMODES semantics)** — Confirmed via ircv3.net / irc.org 005 docs:
  Type A = list (always param), Type B = always param, Type C = param only
  when set, Type D = flag (never param). The claim that the batcher can't
  compute param counts without parsing this stands.
- **§3 (multi-prefix NAMES ordering)** — Confirmed via IRCv3 multi-prefix
  spec: prefixes MUST be in rank order, highest to lowest. Example from
  spec: `353 guest = #tethys :~&@%+aji`. Our parser (see §A.3 below) does
  not handle this at all.
- **§5 (STS)** — Confirmed via IRCv3 STS spec: "Once a client has connected
  securely, and it has verified that an STS persistence policy is in place,
  it then MUST only use a secure transport ... until the policy expires."
  Hexbot's lack of STS handling is a real downgrade risk.
- **§8 / §11 (NOTICE automatic-reply rule)** — Confirmed via RFC 2812
  §3.3.2: "automatic replies MUST NEVER be sent in response to a NOTICE
  message. The object of this rule is to avoid loops between clients
  automatically sending something in response to something it received."
  Hexbot correctly parses commands only in `onPrivmsg`. The requirement
  that CTCP _replies_ use NOTICE (not PRIVMSG) is RFC-backed; still worth
  a grep to confirm `ctcpReply()` actually uses NOTICE.
- **account-tag exists as a real capability** and populates a `@account=`
  tag on every incoming PRIVMSG from an identified user, so the hostmask-vs-
  account critique in §7 is backed by a concrete mechanism that should
  already be reaching hexbot (see A.2).

### A.2 Correction — §5 IRCv3 capability request list

**Original claim:** "Hexbot only flips `enable_chghost`; we should explicitly
request multi-prefix, userhost-in-names, away-notify, account-tag, server-time,
message-tags, batch."

**Reality:** irc-framework `src/commands/handlers/registration.js` already
auto-requests on every connection:

```js
let want = [
  'cap-notify',
  'batch',
  'multi-prefix',
  'message-tags',
  'draft/message-tags-0.2',
  'away-notify',
  'invite-notify',
  'account-notify',
  'account-tag',
  'server-time',
  'userhost-in-names',
  'extended-join',
  'znc.in/server-time-iso',
  'znc.in/server-time',
];
```

So 10 of the 11 caps I told hexbot to request are already being requested
automatically. Hexbot additionally opts into `chghost` via
`enable_chghost: true`. The finding therefore **shifts**:

- The problem is not that hexbot fails to _request_ these caps — it's that
  hexbot **doesn't consume several of them** even though they arrive:
  - **`away-notify`**: no handler in `channel-state.ts`; user `away` state is
    not tracked. Plugins can't ask "is this user away?".
  - **`account-tag`**: `irc-bridge.ts:onPrivmsg` reads `nick/ident/hostname/
target/message` but does not read `event.tags?.account`. That tag is the
    _canonical_ source of truth for who sent a message; hexbot should use it
    to fast-path account lookups and skip NickServ ACC round-trips entirely
    for any message that carries the tag.
  - **`server-time`**: no consumer; log/relay plugins use wall-clock instead
    of the server-provided timestamp, so replayed chathistory or bouncer
    backlog will be mistimed.
  - **`batch`**: no consumer; multi-line batches (e.g. chathistory, netsplit
    QUIT bundles) are processed as if they were unrelated events.
- Still genuinely missing (not auto-requested, need explicit opt-in or
  `requestCap()`):
  - **`echo-message`** — gated by `enable_echomessage: false` default.
  - **`labeled-response`** — not in default list.
  - **`setname`** — gated by `enable_setname: false` default.
  - **`draft/chathistory`** — not in default list.
  - **`sts`** — not in default list; must be consumed at CAP LS / reconnect
    time, not simply "requested" (it's server-advertised).

**Updated §5 todo:**

- [ ] Hard error at startup if `sasl_mechanism === 'PLAIN'` and
      `tls !== true`. _(unchanged, still valid)_
- [ ] Add a handler for the irc-framework `away` event in `channel-state.ts`;
      track per-user away state and emit `channel:awayChanged`.
- [ ] In `irc-bridge.ts:onPrivmsg`, read `event.tags?.account` when
      `account-tag` is active and stash it on `HandlerContext.account` so
      the dispatcher's NickServ-ACC fast path uses the tag before querying
      services.
- [ ] Decide whether to consume `server-time` for logging/relay plugins, or
      document that we intentionally ignore it.
- [ ] Decide whether to enable `enable_echomessage` (needed if we want
      labeled-response dedup) and `enable_setname`.
- [ ] STS: persist directive and bounce to TLS on reconnect; refuse
      downgrade during the policy window. _(unchanged)_
- [ ] Drop the "request `multi-prefix`, `userhost-in-names`, `away-notify`,
      `account-tag`, `server-time`, `message-tags`, `batch`" item — already
      handled upstream.

### A.3 New finding — §3 NAMES reply drops all user prefix modes (P0)

Triggered by the §5 verification: if irc-framework auto-requests
`multi-prefix` and parses the NAMES reply into `user.modes` as an **array**
of mode chars, then hexbot's receiver had better accept an array.

It doesn't. `src/core/channel-state.ts:350`:

```ts
const modes = this.parseUserlistModes(typeof u.modes === 'string' ? u.modes : undefined);
```

And `src/core/channel-state.ts:502-510`:

```ts
private parseUserlistModes(modes: string | undefined): string[] {
  if (!modes) return [];
  const result: string[] = [];
  // irc-framework uses symbols: @ = op, + = voice, % = halfop
  if (modes.includes('o') || modes.includes('@')) result.push('o');
  if (modes.includes('v') || modes.includes('+')) result.push('v');
  if (modes.includes('h') || modes.includes('%')) result.push('h');
  return result;
}
```

Verified via `kiwiirc/irc-framework/src/commands/handlers/channel.js` — the
RPL_NAMEREPLY handler walks `network.options.PREFIX` and pushes an **array**
of mode chars onto `user.modes`. Hexbot's `typeof u.modes === 'string'`
guard always fails, so `undefined` is passed to `parseUserlistModes`, which
returns `[]`.

**Consequence — P0:** Every user observed via NAMES is tracked with **zero
modes** until a subsequent MODE change fires. That means:

- On join to a populated channel, hexbot believes **nobody** has `+o`, `+v`,
  `+h`, `+a`, or `+q`.
- Any plugin that gates behaviour on "is this user an op?" (protection
  plugins, kick enforcement, greeter skipping existing ops, etc.) gets
  wrong answers for the entire pre-existing userlist.
- Mode changes after join eventually fix individual users, but a user who
  never has their mode changed while hexbot is present stays mis-tracked
  forever.

Secondary issue: even if the string-vs-array mismatch were fixed, the
fallback parser in `parseUserlistModes` only recognizes `o/v/h` (and
symbols `@/+/%`). It drops:

- **`a` (admin, `&`)** on InspIRCd/Unreal/OFTC (channel admin).
- **`q` (owner, `~`)** on InspIRCd/Unreal (channel founder).

So on any major modern IRCd outside of Solanum/Libera, even a working
parser would lose founder/admin information.

**Severity:** P0. This is silently wrong on every network hexbot connects
to and directly contradicts the multi-prefix cap that irc-framework goes
out of its way to request on our behalf.

**New §3 todo:**

- [ ] Accept `u.modes` as `string[]` (the actual irc-framework shape);
      drop the `typeof === 'string'` guard. Consider just copying
      `Array.isArray(u.modes) ? u.modes.slice() : []` and letting the
      `PREFIX` set from ISUPPORT tell us which chars are valid user-mode
      flags.
- [ ] Extend the recognized prefix-mode set to at least `q/a/o/h/v` —
      or better, derive from ISUPPORT `PREFIX` instead of hardcoding.
- [ ] Add a channel-state test: feed a NAMES reply with
      `~foo &bar @+baz %qux +quux` and assert the parsed modes are
      `{foo:[q], bar:[a], baz:[o,v], qux:[h], quux:[v]}`.
