# Security Audit: hexbot (full codebase)

**Date:** 2026-04-11
**Scope:** Every `.ts` file in `src/` and `plugins/` (77 files, ~18k LOC)
**Baseline:** `docs/SECURITY.md`

## Summary

hexbot's core security posture is strong. Input is sanitized at the IRC ingress boundary, SQL is universally parameterized, the dispatcher correctly gates handlers on flags before invoking them, NickServ verification is awaited and fails closed, handler errors are contained, timers and message queues are bounded, and no `eval()`/`new Function()` use exists anywhere in the tree. Plugins are forbidden from reading `process.env` and cannot reach `src/` directly — the only isolation escape found is shallow `Object.freeze` on nested plugin config.

The highest-leverage issues cluster in the bot-link subsystem, which is the largest and youngest attack surface. Two are direct violations of rules stated in `docs/SECURITY.md`: a plain `!==` password-hash compare and a `findByHostmask` that returns the first match without specificity ordering. Neither is trivially exploitable on a default deployment, but both should be fixed before hardening is claimed.

**Findings:** 0 critical, 6 warning, 6 info

## Findings

### [WARNING] Password hash compared with `!==` (not timing-safe)

**File:** `src/core/botlink-hub.ts:730`
**Category:** Credentials / botlink auth

```ts
// Auth check — password field is NEVER logged
if (password !== this.expectedHash) {
```

`password` is the `scrypt:<hex>` string the leaf sends in `HELLO`. The hub compares it against a pre-computed expected hash using JavaScript's `!==`, which short-circuits on the first differing byte. On a LAN or loopback deployment (the documented supported topology), the wall-clock difference between a 1-byte and a 40-byte prefix match is measurable over enough samples, leaking the expected hash byte-by-byte. Because the leaf sends the hash directly — the hub never re-hashes — a leaked `expectedHash` is immediately replayable as the `HELLO` password. That is the full botnet compromise described in SECURITY.md §9.

Rate limiting (5 failures / 60s, escalating ban) slows this but doesn't eliminate it: the tracker unbans and a distributed attacker can spread samples across IPs.

**Remediation:** Use constant-time comparison with length guard:

```ts
import { timingSafeEqual } from 'node:crypto';

const a = Buffer.from(password, 'utf8');
const b = Buffer.from(this.expectedHash, 'utf8');
if (a.length !== b.length || !timingSafeEqual(a, b)) {
  // ... existing failure path
}
```

### [WARNING] `findByHostmask` returns first match, not most specific

**File:** `src/core/permissions.ts:198-207`
**Category:** Permissions / identity

```ts
findByHostmask(fullHostmask: string): UserRecord | null {
  for (const record of this.users.values()) {
    for (const pattern of record.hostmasks) {
      if (wildcardMatch(pattern, fullHostmask, true)) {
        return record;
      }
    }
  }
  return null;
}
```

This directly violates `docs/SECURITY.md` §3.3: _"Flag checking must not short-circuit on the first matching hostmask if that hostmask belongs to a different user."_ The method iterates `this.users` in Map insertion order and returns the first user whose pattern matches.

**Exploit scenario:** Admin adds user A with a broad pattern (e.g. `*!*@trusted.example`) and grants it `+o`. Later, admin adds user B with `alice!alice@trusted.example` and grants only `+-`. `alice!alice@trusted.example` matches _both_ patterns. Because A was inserted first, Alice is identified as A and inherits `+o`. Admin intended the opposite — the more specific entry to win.

**Remediation:** Score candidates by specificity (literal characters minus wildcards, or pattern length ignoring `*`/`?`) and return the highest-scoring match. When ties occur, prefer the pattern that matches more of `host` over `ident` over `nick`. Optionally emit a `[security]` warning when two distinct records match the same hostmask on any lookup.

### [WARNING] Bot-link scrypt uses a hardcoded, globally-shared salt

**File:** `src/core/botlink-protocol.ts:50`
**Category:** Credentials / botlink auth

```ts
export function hashPassword(password: string): string {
  const key = scryptSync(password, 'hexbot-botlink-v1', 32);
  return 'scrypt:' + key.toString('hex');
}
```

Every hexbot installation on earth derives its botlink hash with the same literal salt `'hexbot-botlink-v1'`. Consequences:

1. **Shared rainbow table.** An attacker can precompute scrypt hashes for a dictionary once and reuse the table against every hexbot botnet they encounter. Scrypt's memory-hardness is amortized, not per-target.
2. **Cross-installation replay.** If two botnets happen to share a password (common for defaults, test configs, or reused secrets), a captured hash from one works against the other.
3. **Combined with the `!==` issue above**, an attacker who extracts the hash needs only that hash to connect — never the plaintext.

**Remediation:** Derive the salt from a per-installation secret (e.g. random bytes written to `data/botlink.salt` on first run, `chmod 600`), and include it in the `HELLO` frame alongside the hash so peers can agree on it. Alternatively, require a minimum password entropy (≥32 random bytes) and document that weaker passwords are unsafe. Rotate existing bot-link passwords after applying the fix since old hashes were computed under the shared salt.

### [WARNING] PluginAPI config is frozen shallowly — nested objects and arrays are mutable

**File:** `src/plugin-loader.ts:436-453, 497-498, 521`
**Category:** Plugin isolation

```ts
const pluginBotConfig: PluginBotConfig = {
  irc: {
    ...this.botConfig.irc,
    channels: this.botConfig.irc.channels.map((c) => (typeof c === 'string' ? c : c.name)),
  },
  owner: { ...this.botConfig.owner },
  identity: { ...this.botConfig.identity },
  services: { type, nickserv, sasl },
  logging: { ...this.botConfig.logging },
  chanmod: this.botConfig.chanmod ? { ...this.botConfig.chanmod } : undefined,
};
// ...
botConfig: Object.freeze(pluginBotConfig),
config: Object.freeze({ ...config }),
// ...
return Object.freeze(api);
```

`Object.freeze` is shallow. A plugin can mutate nested state through the frozen handle:

```ts
api.botConfig.irc.channels.push('#pwn'); // works — channels is a plain array
api.botConfig.identity.require_acc_for.pop(); // works — nested array
api.botConfig.services.nickserv = 'Evil'; // works — nested object is not frozen
```

Whether this escalates depends on who reads the mutated field later. Plugin A mutating `api.botConfig` doesn't affect the real `this.botConfig`, but other plugins will see the mutation because `pluginBotConfig` is shared by reference across all plugin contexts built from it. One misbehaving plugin can silently reshape another plugin's view of the bot config.

**Remediation:** Add a `deepFreeze` helper and apply it to `pluginBotConfig`, the merged plugin `config`, and the nested API subobjects (`permissions`, `services`, etc.) at construction time. Build a fresh `pluginBotConfig` per plugin rather than sharing a single object across plugins.

### [WARNING] Incoming sync frames have no per-field size cap

**File:** `src/core/botlink-sync.ts:46-66`, `src/core/botlink-sharing.ts` (CHAN_BAN_SYNC apply path)
**Category:** DoS / botlink trust boundary

```ts
static applyFrame(frame: LinkFrame, channelState: ChannelState): void {
  if (frame.type !== 'CHAN') return;
  channelState.injectChannelSync({
    // ...
    users: isObjectArray(frame.users)
      ? frame.users.map((u) => ({ ... }))
      : [],
  });
}
```

The protocol layer enforces a 64 KB frame cap (`botlink-protocol.ts`), but an inbound `CHAN` or `CHAN_BAN_SYNC` frame within that budget can still carry thousands of entries. A hub compromise or a malicious hub pretending to sync fills leaf memory with user/ban objects. The trust model places the hub as authoritative, but defense in depth is cheap here and the rule in SECURITY.md §9 (validate all frames) argues for applying it.

**Remediation:** Add explicit bounds when applying sync frames (e.g. `MAX_USERS_PER_CHAN_FRAME = 5000`, `MAX_BANS_PER_SYNC = 2000`). Drop overflow with a `[security]` warning. Apply the same cap to `ADDUSER` fan-outs that carry `hostmasks` and `channels`.

### [WARNING] Bot-link hub defaults to `0.0.0.0` without a non-loopback warning

**File:** `src/core/botlink-hub.ts:189`, `config/bot.example.json`
**Category:** Secure defaults

```ts
host = this.config.listen?.host ?? '0.0.0.0',
```

SECURITY.md §9 mandates that bot-link TCP should be bound to a private IP or `127.0.0.1` when bots are co-located, and notes that the protocol is unencrypted. The code defaults to every interface without a startup log that would alert an operator who pasted `config/bot.example.json` unchanged. The only protection when bound to a public IP is the bot-link password — subject to the two WARNING findings above.

**Remediation:** Default to `127.0.0.1`. On startup, if `listen.host` is `0.0.0.0` / `::` / a non-loopback address and `auth_ip_whitelist` is empty, log a prominent `[security]` warning with the text from SECURITY.md §9 and refuse to start unless an explicit `listen.acknowledge_public = true` flag is set.

### [INFO] Permission command announcements do not strip IRC formatting from user-supplied handles

**File:** `src/core/commands/permission-commands.ts:33, 55, 115, 119`
**Category:** Output safety

```ts
ctx.reply(`User "${handle}" added with hostmask ${hostmask} and flags ${flags}`);
ctx.reply(`User "${handle}" removed`);
ctx.reply(`Channel flags for "${handle}" in ${channel} set to "${flagsArg}"`);
ctx.reply(`Global flags for "${handle}" set to "${flagsArg}"`);
```

SECURITY.md §5.2 requires `api.stripFormatting(...)` on user-controlled strings in permission grant/revoke announcements. These handlers only run for `+n` users, so the practical spoof window is small, but the rule is explicit.

**Remediation:** Wrap `handle`, `hostmask`, and `flagsArg` in `api.stripFormatting(...)` (or the core equivalent) in all four call sites.

### [INFO] Bot-link auth tracker can grow slowly under a pacing scanner

**File:** `src/core/botlink-hub.ts:847-874` (`sweepStaleTrackers`)
**Category:** DoS / memory growth

The cleanup requires both `banExpired` and `failureWindowExpired` before evicting a tracker entry with `banCount === 0`. An attacker that sends one failed auth per IP just before `auth_window_ms` expires keeps `firstFailure` rolling forward, so `failureWindowExpired` never becomes true and the entry is never evicted. Memory grows linearly with unique attacker IPs over the lifetime of the process.

Impact is low in practice: at ~100 bytes per entry, a year of persistent scanning tops out in the low megabytes, and the real-world scanner pattern (sprays then gives up) mostly self-cleans. Worth fixing for peace of mind.

**Remediation:** Evict an entry whose `failures` count sits at zero for more than `auth_window_ms`, regardless of `firstFailure`. Or: age-out entries past a hard TTL (e.g. 7 days) unconditionally.

### [INFO] `getModLog` builds SQL via template-literal WHERE concatenation

**File:** `src/database.ts:185`
**Category:** Protocol injection / defense in depth

```ts
const sql = `SELECT id, timestamp, action, channel, target, by_user AS "by", reason FROM mod_log ${where} ORDER BY id DESC ${limit}`;
return db.prepare(sql).all(...params) as ModLogEntry[];
```

The `where` and `limit` fragments are assembled from hardcoded predicate strings — all real user values flow through `?` placeholders and `params`. Not currently exploitable, and `getModLog` isn't exposed to IRC or plugins. Flagged because the pattern is fragile: a future refactor that interpolates a filter value would introduce SQL injection invisibly.

**Remediation:** Prepare a single statement with `COALESCE`-based predicates (`(? IS NULL OR action = ?)`) or switch to a tiny query-builder. Either eliminates the template literal.

### [INFO] `.part` reason message is not sanitized before handing to `irc-framework`

**File:** `src/core/commands/irc-commands-admin.ts:93`
**Category:** Input validation / defense in depth

```ts
const message = parts.slice(1).join(' ') || undefined;
client.part(channel, message);
```

`irc-framework`'s `.part()` handles PART framing correctly, so this is not an injection vector today. But every sibling call site (`join`, `kick`, `topic`, etc. in `src/core/irc-commands.ts`) runs `sanitize()` on user-supplied strings first; this one doesn't. Apply the same hygiene for consistency.

**Remediation:** `const message = sanitize(parts.slice(1).join(' ')) || undefined;`

### [INFO] CMD and PARTY_CHAT rate-limit violations diverge silently

**File:** `src/core/botlink-hub.ts:897-910`
**Category:** Botlink protocol consistency

CMD over-limit sends `ERROR:RATE_LIMITED` back to the leaf; PARTY_CHAT silently drops. Not a vulnerability, but the inconsistency makes it harder for a leaf to detect throttling, and the silence helps attackers tune their rate. Pick one convention and document it.

### [INFO] `Permissions.loadFromDb` has no UserRecord schema validation

**File:** `src/core/permissions.ts:287-302`
**Category:** Input validation / DB integrity

`JSON.parse` is wrapped in try/catch, but a successfully-parsed record with the wrong shape (e.g. `hostmasks: "not-an-array"`) lands in memory and only explodes when a later lookup iterates it. The database file is owner-controlled, so this is mostly a stability concern rather than a security one — but a corrupted row shouldn't be able to take the permissions subsystem down mid-flight.

**Remediation:** Add a structural check (array guards, string checks, flag-character whitelist) when loading each row. Skip and log rows that fail.

## Passed checks

The following categories were audited against SECURITY.md and found clean:

- **IRC ingress sanitization** (`src/irc-bridge.ts`): every event handler runs nick/ident/host/target/text through `sanitize()` before constructing `HandlerContext`.
- **`raw()` discipline** (`src/core/irc-commands.ts`): all `raw()` callers wrap user-supplied values in `sanitize()`; typed `.say`/`.notice`/`.mode` used elsewhere.
- **CTCP handling** (`plugins/ctcp`, `src/irc-bridge.ts`): double-sanitized; rate-limited to 3 responses per nick per 10s.
- **SQL parameterization**: every data-path query uses prepared statements with `?` placeholders (`database.ts`, `admin-list-store.ts`, `ban-store.ts`, plugin DB calls). One construction pattern flagged above for style.
- **`eval` / `new Function` / `vm`**: zero matches across `src/` and `plugins/`.
- **Dispatcher flag gating** (`src/dispatcher.ts:282-307`): flag check runs before `entry.handler(ctx)`; handler wrapped in try/catch; async results awaited.
- **NickServ race** (`services.ts`, `plugins/chanmod/auto-op.ts:102-119`): `require_acc_for` triggers `await api.services.verifyUser(nick)` before op; 5s timeout resolves `verified: false` (fails closed).
- **Hostmask matching case-insensitive** (`permissions.ts:201`, `wildcard.ts`): `wildcardMatch(..., true)`; uses `ircLower` with configured casemapping.
- **Insecure hostmask warning** (`permissions.ts:370-386`): `nick!*@*` on a user with `n`/`m`/`o` emits a `[security]` warning on add.
- **Owner flag implication** (`permissions.ts:335-353`): `n` short-circuits to allow before any specific flag check.
- **REPL logging & isolation** (`src/repl.ts`): every command logged at `info`, announced via DCC, executed with `source: 'repl'`. Reads only from `process.stdin` — no network surface.
- **Client-only IRCv3 tags**: no call site uses `ctx.tags['+foo']` for a trust decision; the account lookup uses the live `account-notify` map in `dispatcher.ts:316-329`.
- **Mode-enforce loop prevention** (`plugins/chanmod/mode-enforce.ts`): cooldown + `markIntentional` + debounce prevents self-triggered cycles.
- **Plugin API direct `src/` imports**: none (plugins use `import type` only).
- **Plugin `process.env` access**: no direct reads in any plugin; `resolveSecrets` runs before `init()` and the api surface exposes no `env` handle.
- **Plugin teardown & timer cleanup** (`plugin-loader.ts:320-365`, `dispatcher.ts:263-273`): `teardown()` awaited and wrapped; `unbindAll(pluginId)` clears `setInterval` handles for that plugin; dispatcher timers held in a map keyed by bind entry.
- **Message queue bounds** (`src/core/message-queue.ts`): `MAX_DEPTH = 500`, excess dropped with warning; token-bucket rate limit (default 2 msg/sec + 4 burst) using integer ms arithmetic.
- **ReDoS**: no `new RegExp(userInput)`; all literal regexes are simple character classes or anchored alternations.
- **Channel/botlink memory maps**: `networkAccounts`, `authTracker` (with caveat above), `pendingHandshakes`, `remotePartyUsers`, `activeRelays` are cleaned on QUIT / leaf-disconnect / expiry.
- **Reconnect backoff** (`connection-lifecycle.ts:84-159`, `botlink-leaf.ts:422-433`): exponential with jitter, capped, bounded retries.
- **Scrypt cost**: Node default `N=16384, r=8, p=1` — ~100-200ms/hash, reasonable for this use case (handshake, not high-frequency).
- **Botlink frame sanitization** (`botlink-protocol.ts:54-72`): `sanitizeFrame` recursively strips `\r\n\0` from all string fields including nested objects and arrays.
- **Botlink frame size cap** (`botlink-protocol.ts:164-169`): 64 KB enforced before JSON parse on inbound; also on outbound.
- **Botlink handshake timer**: always cleared in `onClose`; no dangling timers.
- **Botlink hub-only frame fan-out**: `HUB_ONLY_FRAMES` set excludes `CMD`/`RELAY_*`/`PROTECT_ACK`/`ADDUSER`/`SETFLAGS`/`DELUSER` from broadcast; `fromBot` is overwritten to the authenticated botname at `botlink-hub.ts:887` before fan-out.
- **DCC race mitigations** (`src/core/dcc.ts`): 30s listener timeout, `server.once('connection', ...)` + `server.close()`, timer cleared on accept, port released on both paths, session limit enforced, permissions checked before offer, optional NickServ verification.
- **Config world-readable refusal** (`bot.ts:566-578`): `stat.mode & 0o004` → exit with `chmod 600` instruction.
- **`validateResolvedSecrets`** (`config.ts:327-351`): called at startup; throws with exact env var name on missing required secret.
- **Example configs**: `config/bot.example.json` + `config/plugins.example.json` contain only placeholder `*_env` references; no real credentials.
- **`.gitignore`**: covers `.env`, `.env.*`, `config/*.json` (except `*.example.json`), `*.db`, `*.db-*`.
- **Secrets in logs**: no logger call interpolates `config.services.password`, `config.botlink.password`, or `config.proxy.password`.
- **Output safety**: `split-message.ts` caps at 400 bytes/line, splits on last space, limits to 4 lines + `...`. `stripFormatting` regex covers the full SECURITY.md §5.2 control-char list including color parameters. Exposed via `api.stripFormatting`.
- **Logger** (`src/logger.ts`): uses Node `util.format`; no raw interpolation of user strings; chalk handles ANSI safely.
- **Secure defaults**: hostmask identity, `require_acc_for: ['+o','+n']`, SASL on, TLS on, admin commands `+n`, help `-`, plugin `permissions` read-only.

## Recommendations

### Before next deployment

1. **Fix `!==` password compare** (`botlink-hub.ts:730`) — one-line change to `timingSafeEqual`.
2. **Fix `findByHostmask` specificity ordering** (`permissions.ts:198-207`) — direct SECURITY.md §3.3 violation.
3. **Deep-freeze `pluginBotConfig`** (`plugin-loader.ts:497`) and build a per-plugin copy to contain cross-plugin mutation.

### Before feature freeze

4. **Per-install bot-link salt** or mandatory high-entropy password + SECURITY.md note that old hashes must be rotated after the change.
5. **Bound sync-frame array sizes** (`botlink-sync.ts`, `botlink-sharing.ts`).
6. **Default `botlink.listen.host` to `127.0.0.1`** and emit a loud startup warning when bound non-loopback without an ack flag.
7. **Apply `stripFormatting` to handles** in `permission-commands.ts`.

### Opportunistic

8. Rework `getModLog` SQL construction to eliminate template literals.
9. Sanitize the `.part` message for consistency.
10. Make `PARTY_CHAT` rate-limit behavior match `CMD` (send `ERROR:RATE_LIMITED` rather than silently drop) or document the divergence.
11. Tighten `Permissions.loadFromDb` with a structural UserRecord check.
12. Evict zero-failure auth tracker entries past a hard TTL regardless of rolling window.

### Documentation

- Add a brief note to SECURITY.md §9 clarifying that a relay session on the target bot runs with the _hub_'s permissions, not the target's — this is intentional per the trust model but was not obvious while reading the code.
- Document the per-install salt rotation procedure once #4 lands.
