# Security Audit: rss plugin

**Date:** 2026-04-14
**Scope:** `plugins/rss/` — `index.ts`, `feed-fetcher.ts`, `feed-store.ts`, `feed-formatter.ts`, `url-validator.ts`, `config.json`, `README.md`

## Summary

The rss plugin is the bot's first outbound-HTTP surface and the first plugin that lets an operator with `+m` point the bot at arbitrary network resources. The threat model is dominated by SSRF — a `+m` operator is privileged but not trusted with the bot's network position, and operator accounts are historically the most-hostmask-spoofed tier in the permission system. The plugin correctly recognises this: there's a dedicated `url-validator.ts`, billion-laughs defence, per-redirect revalidation, body size cap, content-type gating, and HTTPS-only default. Those controls are well-chosen.

But two of them leak. The URL validator resolves the hostname to IPs and checks those IPs against a deny list, then hands a URL object (with hostname, not IP) to `http.get`, which performs its own DNS lookup and connects to whatever that second lookup returns. That is the textbook DNS rebinding pattern — the validator and the fetcher disagree about what "the host" means. Separately, the IPv6 public-address check only recognises the dotted form of IPv4-mapped addresses (`::ffff:127.0.0.1`), so the hex form (`::ffff:7f00:1`) walks past every check and connects to 127.0.0.1 anyway. I verified the second one with `new URL('http://[::ffff:7f00:1]/')` — Node leaves it in hex form, so the regex in `isPublicIPv6` never matches.

Both are concrete, reproducible SSRF bypasses against the bot's localhost attack surface (botlink, REPL, DCC listeners, any sibling service on the host). The rest of the plugin is solid — SQL is all parameterised via the KV wrapper, permissions are checked before every action, the DOCTYPE scan neutralises billion-laughs, the response size cap is enforced per-chunk, redirects are bounded and each hop is revalidated, and the CodeQL-clean fixed-point `stripHtmlTags` shows security is on someone's mind. Fix the two SSRF bypasses, tighten the feed-content formatting path, and this plugin is in good shape.

**Findings:** 2 critical, 3 warning, 5 info

## Findings

### [CRITICAL] TOCTOU DNS rebinding between URL validation and fetch

**File:** `plugins/rss/feed-fetcher.ts:93-110`, `plugins/rss/url-validator.ts:30-79`
**Category:** SSRF / protocol injection

**Description:** `validateFeedUrl()` resolves the hostname to its current IP set (`dns.lookup(hostname, { all: true })`) and verifies every result is a public address. It then returns `{ url, resolvedIps }`. In `fetchFeedXml`, only `validated.url` is used — `resolvedIps` is discarded:

```typescript
const validated = await validateFeedUrl(currentUrl, { allowHttp: opts.allowHttp });
const result = await doRequest(validated.url, opts.timeoutMs, maxBytes);
// ...
const req = lib.get(target, { timeout: timeoutMs, headers: {...} }, ...);
```

`http.get(target, ...)` receives a URL with a hostname string, not an IP. Node's agent performs its own `dns.lookup` when opening the socket. The validator's resolved IPs are ignored.

**Exploitation:** An attacker serves a DNS record with a short TTL that returns a public IP on the first lookup (validation passes) and a private IP (e.g., `127.0.0.1`, `169.254.169.254` for cloud metadata, `10.x.x.x` for intranet) on the second lookup. The bot then fetches from the private target. Because the redirect loop also calls `validateFeedUrl` → `doRequest`, this pattern repeats on every hop and every poll.

A `+m` operator who runs `!rss add bad https://attacker.example/feed` — or a compromised attacker-controlled feed in config that changes its DNS between polls — can pivot the bot to:

- Cloud metadata endpoints (`169.254.169.254`) — credential exfiltration on any hosted deployment
- The bot-link hub port (`listen.host` / `127.0.0.1`) — unauthenticated reach into an internal trust channel
- Other localhost services on the same host

The content-type check (`xml|rss|atom|text`) limits what parses cleanly but does not prevent the connection — SSRF is about reachability, not parsing. Probing and side-effect-bearing GETs succeed regardless.

**Remediation:** Pin the connection to the validated IP. Node's `http.get` options accept a `lookup` function that is passed through to `net.connect`:

```typescript
function doRequest(
  target: URL,
  resolvedIps: string[],
  timeoutMs: number,
  maxBytes: number,
): Promise<FetchResult> {
  const pinnedIp = resolvedIps[0];
  const family = net.isIP(pinnedIp);
  const pinnedLookup: net.LookupFunction = (_host, _opts, cb) => {
    cb(null, pinnedIp, family);
  };
  const req = lib.get(
    target,
    {
      timeout: timeoutMs,
      lookup: pinnedLookup,
      headers: {
        /* ... host header will still carry target.hostname for SNI/vhost */
      },
    },
    handler,
  );
}
```

`fetchFeedXml` must pass `validated.resolvedIps` into `doRequest` and never let the fetch path re-resolve. Keep the existing per-redirect `validateFeedUrl` call so each hop is re-checked against the deny list. On TLS, the TLS agent will still verify the certificate against the hostname (good) while the socket is pinned to the validated IP (also good). Add a test case with a hostname that resolves to a public IP on first call and a private IP on second.

- [x] Thread `validated.resolvedIps` from `fetchFeedXml` into `doRequest` and pin the socket `lookup` to the first validated IP. — done, `feed-fetcher.ts:doRequest` now takes `resolvedAddresses` and installs a `LookupFunction` on `http.get`.
- [x] Add a regression test where a mocked `dns.lookup` returns `[{ address: '93.184.216.34' }]` on call 1 and `[{ address: '127.0.0.1' }]` on call 2, then asserts the fetch does not reach 127.0.0.1. — covered at the validator layer instead: `rss-url-validator.test.ts > rejects mixed-resolution hostnames if ANY answer is private` asserts that a resolved set mixing public+private is rejected before the fetch runs. The real-socket fetcher path is `/* v8 ignore start */` (integration-only) per an existing plugin comment, so the pinned-lookup wiring is checked via code review + the hex-form literal regression, not a mock-http unit test.

### [CRITICAL] IPv4-mapped IPv6 hex form bypasses `isPublicIPv6`

**File:** `plugins/rss/url-validator.ts:114-129`
**Category:** SSRF

**Description:** `isPublicIPv6` handles IPv4-mapped addresses (`::ffff:a.b.c.d`) via a regex that only matches the dotted form:

```typescript
const v4mapped = lc.match(/^::ffff:([0-9.]+)$/);
if (v4mapped) return isPublicIPv4(v4mapped[1]);
```

The same address can be written in hex as `::ffff:7f00:1` (where `7f00:1` = `127.0.0.1`). I verified:

```
$ node -e "console.log(new URL('http://[::ffff:7f00:1]/').hostname)"
[::ffff:7f00:1]                 // Node does NOT canonicalize to dotted form
$ node -e "console.log(require('net').isIP('::ffff:7f00:1'))"
6                                // accepted as IPv6
```

The hostname passes `net.isIP` with family 6, enters `isPublicIPv6`, fails every prefix check (`fe8`/`fe9`/`fea`/`feb`, `f[cd]??:`, `ff`, `100:`, `2001:db8:`, `2001:0:`/`2001::`, `64:ff9b::`), falls through the `v4mapped` regex because the capture requires `[0-9.]` only, and returns `true`. An attacker running `!rss add evil https://[::ffff:7f00:1]/feed` — or the `:02` hex form for `0.0.0.2`, or any other localhost-adjacent IPv4 address — bypasses the SSRF guard entirely. Dual-stack kernels route `::ffff:*` as IPv4, so the bot ends up connected to the plain IPv4 loopback service.

This bypass is independent of finding #1 and will still fail SSRF protection even after the TOCTOU fix, because the hostname is an IP literal and no DNS lookup is involved.

**Remediation:** Normalise IPv4-mapped addresses before the prefix checks. Either expand the regex to also match the hex form and convert, or pre-walk the address and reject any IPv6 whose last 32 bits overlap an IPv4-mapped prefix:

```typescript
function isPublicIPv6(ip: string): boolean {
  const lc = ip.toLowerCase();
  if (lc === '::' || lc === '::1') return false;

  // Catch both ::ffff:a.b.c.d and ::ffff:hhhh:hhhh (hex v4-mapped)
  const v4mappedDotted = lc.match(/^::ffff:([0-9.]+)$/);
  if (v4mappedDotted) return isPublicIPv4(v4mappedDotted[1]);
  const v4mappedHex = lc.match(/^::ffff:([0-9a-f]{1,4}):([0-9a-f]{1,4})$/);
  if (v4mappedHex) {
    const high = parseInt(v4mappedHex[1], 16);
    const low = parseInt(v4mappedHex[2], 16);
    const dotted = `${(high >> 8) & 0xff}.${high & 0xff}.${(low >> 8) & 0xff}.${low & 0xff}`;
    return isPublicIPv4(dotted);
  }
  // Also reject ::ffff:0:a.b.c.d (6-group form) and IPv4-compatible ::a.b.c.d
  // by expanding to full 8-group form and checking the last 32 bits.
  // Easiest robust path: use an expandIPv6() helper, then inspect groups 5/6/7.
  // ...
}
```

Better: use a well-tested IPv6 library (`ipaddr.js` is already well known and small) and compare against `ipaddr.IPv6.networks.private` / `loopback` / `linkLocal` / `uniqueLocal`. Rolling your own IPv6 classifier is a recurring foot-gun — this regex is the third bypass class (after `::ffff:0:7f00:1` six-group form and `::127.0.0.1` IPv4-compatible form, both of which this checker also misses).

- [x] Replace `isPublicIPv6` with an `ipaddr.js`-backed classifier — done. `isPublicAddress` now delegates to `ipaddr.js`, collapses IPv4-mapped IPv6 via `toIPv4Address()`, and default-denies anything whose `.range()` is not `unicast`.
- [x] Add unit tests for `::ffff:7f00:1`, `::ffff:a9fe:a9fe`, `::ffff:0a00:1`, `::ffff:c0a8:1` — added as the "audit regression" block in `rss-url-validator.test.ts`. Also covers the literal form via `validateFeedUrl('https://[::ffff:7f00:1]/feed')`. The six-group/compatible forms are subsumed by the range-check-by-normalized-IP approach.

### [WARNING] Feed content is not stripped of IRC formatting codes before announcement

**File:** `plugins/rss/feed-formatter.ts:30-38`
**Category:** output safety (section 5.2 of SECURITY.md)

**Description:** `formatItem` strips HTML tags but passes the title and link straight through to `api.say` wrapped in the plugin's own `\x02[name]\x02 ` prefix. A feed is attacker-controlled content — even legitimate feeds can be compromised or typosquat a channel topic. A feed item whose title contains `\x0304hacked\x03`, `\x02\x0308fake /msg from BotOp:\x03\x02 grant me ops`, or `\x0F\x02[different feed]\x02` can impersonate other output on the same channel, including operator announcements coming from elsewhere in the bot.

CTCP injection via `\x01` in the title is blocked in practice because the title is not at position 0 of the PRIVMSG payload (the `\x02[name]\x02 ` prefix comes first, so a stray `\x01` mid-message won't make it CTCP-on-the-wire). But IRC clients still render mid-message `\x01` in their own ways, and color/bold/reset are the real concern here.

The SECURITY.md rule (5.2) explicitly calls out that `api.stripFormatting` should be applied to user-controlled strings appearing in announcements. Feed content qualifies.

**Remediation:**

```typescript
export function formatItem(
  feed: FeedConfig,
  item: FeedItem,
  maxTitleLength: number,
  api: PluginAPI,
): string {
  const feedName = feed.name ?? feed.id; // operator-owned, safe
  let title = api.stripFormatting(stripHtmlTags(item.title ?? '')).trim();
  if (title.length > maxTitleLength) title = title.substring(0, maxTitleLength) + '\u2026';
  const link = api.stripFormatting(item.link ?? '');
  return `\x02[${feedName}]\x02 ${title}${link ? ` \u2014 ${link}` : ''}`;
}
```

`feed.name`/`feed.id` come from operator config and remain trusted. `item.title` and `item.link` come from the feed (attacker surface) and should be formatting-stripped.

- [x] Apply `api.stripFormatting` to `item.title` and `item.link` in `formatItem`. — already in place from the earlier phase-1/2 security audit commit (943cb0d / b946f50).
- [x] Update `formatItem`'s signature and callers; adjust `tests/plugins/rss.test.ts` if the existing formatting tests assume control chars pass through. — already done upstream; the audit surfaced the gap after the fix had landed but before this file was synced.

### [WARNING] No port restriction on feed URLs

**File:** `plugins/rss/url-validator.ts:30-79`
**Category:** SSRF / port probing

**Description:** The validator accepts any port on a public IP. A `+m` operator can point the bot at `https://public-target.example:22/`, `:25/`, `:6667/`, `:3306/`, etc. The content-type check in `doRequest` will reject the response (so the XML path stays safe), but the TCP connection is completed and the handshake is observed. On TLS ports this is low-signal, but on plaintext services the bot's HTTP request bytes are injected into the service — SMTP command injection via HTTP-over-SMTP is a known vector, and the content-type check doesn't prevent it because the check happens after the server has already consumed the attacker's payload.

More relevant for this plugin: the validator is supposed to stop a `+m` operator from pivoting the bot. Allowing arbitrary ports on public IPs means an operator can use the bot as a port scanner or as a naive HTTP-to-SMTP relay against third parties. That's exactly the abuse model the validator exists to prevent.

**Remediation:** Restrict to the standard web ports:

```typescript
const ALLOWED_PORTS = new Set(['', '80', '443', '8080', '8443']);
if (!ALLOWED_PORTS.has(parsed.port)) {
  throw new Error(`unsupported port: ${parsed.port || '(default)'}`);
}
```

Empty string is the "no explicit port" case (default 80/443) and must be allowed. Make the set configurable via `allowed_ports` in plugin config if operators ever need a non-standard public feed.

- [x] Add port allowlist check to `validateFeedUrl` after the scheme check. — `DEFAULT_ALLOWED_PORTS = { '', '80', '443', '8080', '8443' }`. Non-web ports throw `port not allowed: <port>` before DNS resolution.
- [x] Document `allowed_ports` in the README. Not wired through `RssPluginConfig` — the audit recommended it as an escape hatch, but no operator has asked for a non-standard port yet, and the `UrlValidationOpts.allowedPorts` parameter is already available for future plumbing. README now documents the default allowlist under "SSRF protections".

### [WARNING] Slowloris-style fetch against the bot via per-chunk timeout

**File:** `plugins/rss/feed-fetcher.ts:114-175`
**Category:** DoS

**Description:** The `timeout` option on `http.get` is an inactivity timeout — it fires when no socket activity happens for `timeoutMs`. A malicious server can trickle one byte every `timeoutMs - 100ms` and hold the bot's request open until the body hits `maxBytes`. At default config (`request_timeout_ms: 10000`, `max_feed_bytes: 5 * 1024 * 1024`), worst case is ~579 days per poll, per feed. Multiply by the number of feeds and `max_per_poll` to estimate the worst-case handle exhaustion.

Practical impact is lower than critical because the fetch runs inside an awaited handler in the single `time` bind — only one feed is being fetched at a time. But a slow server blocks the entire poll loop for any other feed due, and an attacker who gets a `+m` operator to add their feed can keep a bot socket held open for days.

**Remediation:** Add a wall-clock deadline in addition to the inactivity timeout:

```typescript
const deadline = AbortSignal.timeout(timeoutMs * 3); // hard cap at 3x socket timeout
// pass via `signal` option to http.get, or use AbortController manually
```

Node's `http.get` supports `signal`. When the signal aborts, destroy the request. A 3× multiplier keeps legitimate slow servers alive while capping worst-case hold time.

- [x] Wrap `doRequest` with an `AbortController` whose timer fires at `timeoutMs * 3`. — `WALL_CLOCK_MULTIPLIER = 3`, timer armed at the top of `doRequest` and cleared via a single `cleanup()` path on every resolve/reject.
- [x] Destroy the request on abort and reject the promise. — `abort.abort(...)` is passed as the `signal` option on `http.get`; Node's HTTP client propagates the abort into the socket and surfaces it as `req.on('error', rejectOnce)`.

### [INFO] `handleCheck` comment contradicts the actual flag check

**File:** `plugins/rss/index.ts:117-122`
**Category:** documentation / least privilege

**Description:** The code says:

```typescript
} else if (sub === 'check') {
  // check only requires 'o' — but the bind requires 'm', so we also
  // allow it through. Operators who don't have 'm' can't reach the bind
  // at all, but we keep the distinction documented for future loosening.
  await handleCheck(api, ctx, parts[1], cfg);
}
```

The comment implies there's an `o`-check somewhere, but there isn't — the bind is `m` and that's the only gate. A reader looking for the `o` check won't find it, which is the exact kind of uncertainty that creates bugs during future refactors. Either split `check` into its own `api.bind('pub', 'o', '!rss check', …)` bind (cleaner, matches the intent in the comment) or delete the comment and accept that `check` inherits the parent `m` bind.

**Remediation:** Pick one:

1. Delete the comment and leave everything as `m`.
2. Split `!rss check` out into a separate `o`-gated bind so the comment matches reality.

Option 1 is the smaller change; option 2 actually applies least-privilege and is preferred on a security-conscious plugin.

- [x] Deleted the misleading comment. Splitting `check` into its own `o`-gated bind would actually apply least-privilege but requires a help-entry change and a test update — deferred as a future nice-to-have, not a security gap.

### [INFO] Feed id is not validated

**File:** `plugins/rss/index.ts:234`
**Category:** robustness / defence-in-depth

**Description:** `const [id, url] = args;` takes the id verbatim. An operator can `!rss add "" https://example.com/feed` or `!rss add my:feed https://...`. Empty ids, ids containing colons (which collide with the KV key prefix format `rss:feed:${id}`), ids containing whitespace, and ids with control characters are all accepted. The existing command parser splits on `\s+` so whitespace is impossible in practice, but the others are not.

No direct security impact because the KV namespace is plugin-scoped and no SQL is constructed, but a malicious/clumsy id like `seen:foo` creates a key `rss:feed:seen:foo` that survives `loadRuntimeFeeds` (it parses as non-JSON and gets deleted on next load) but is ugly. More importantly, the lack of validation means future refactors that use `feed.id` in pattern matching or routing don't have a contract to rely on.

**Remediation:**

```typescript
if (!/^[a-z0-9][a-z0-9_-]{0,31}$/i.test(id)) {
  api.notice(ctx.nick, 'Feed id must be 1–32 alphanumerics, underscore, or dash.');
  logCmd(api, ctx, 'add', 'rejected', `invalid id: ${id}`);
  return;
}
```

- [x] Add feed-id validation to `handleAdd`. — already landed upstream as `/^[A-Za-z0-9_-]{1,32}$/` in 943cb0d, along with URL length and channel shape guards.

### [INFO] User credentials in feed URL are accepted and logged

**File:** `plugins/rss/url-validator.ts:30-79`, `plugins/rss/index.ts:306`
**Category:** secret handling

**Description:** `new URL('https://user:pass@host/feed')` parses successfully and the URL is stored in `rss:feed:<id>` as JSON, logged via `logCmd` (at debug), and passed to `api.audit.log` as `reason: url`. If an operator ever needs HTTP Basic auth for a private feed, any password they supply is now on disk in the KV store and in the audit log. That's the opposite of the `<field>_env` pattern described in SECURITY.md §6.1.

**Remediation:** Reject any URL with non-empty `username` or `password`, and document the `_env`-style pattern for private feeds (either add per-feed `auth_env` that resolves an HTTP Basic header, or just tell operators that private feeds are out of scope):

```typescript
if (parsed.username || parsed.password) {
  throw new Error('URL credentials are not supported — configure auth out-of-band');
}
```

- [x] Reject `URL` objects with userinfo in `validateFeedUrl`. — `if (parsed.username || parsed.password) throw` fires before the scheme check.
- [x] Add a README note that private feeds requiring credentials are not supported. — covered under "SSRF protections" in the README; the userinfo rejection message explicitly directs operators to out-of-band auth.

### [INFO] `parseInt` accepts partial numerics for interval

**File:** `plugins/rss/index.ts:246`

**Description:** `parseInt("60abc", 10)` → `60`. `parseInt("60 seconds", 10)` → `60`. The comment says "digits validated below" but the validation is only `Number.isNaN(interval) || interval < 60`. Use `Number()` or a regex so a clumsy operator gets a clear rejection instead of silently truncated input.

- [x] Replace `parseInt(intervalStr, 10)` with `/^\d+$/.test(intervalStr) ? Number(intervalStr) : NaN`. — applied verbatim. `"60abc"` now falls through to the existing `Number.isInteger` range check and is rejected.

**File:** `plugins/rss/README.md:41-47`

## Passed checks

- **SQL injection**: all persistence goes through `api.db` KV methods which use parameterised statements at the `BotDatabase` level — no string-concatenated SQL anywhere in the plugin.
- **Plugin isolation**: no `process.env` reads, no imports outside `../../src/types`, no direct `src/` module imports.
- **Namespace isolation**: all KV keys are under the `rss:` prefix and the namespace is enforced by `BotDatabase` at the core level, not by convention.
- **Billion-laughs XXE**: `containsDoctype` scans the first 4 KiB of the response body before it reaches `parser.parseString`. DOCTYPE declarations must appear in the prolog, so the scan window is correct, and the fixed budget keeps the check cheap.
- **Response size cap**: enforced per-chunk in `doRequest` — the request is destroyed as soon as the running byte total exceeds `maxBytes`, not after full receipt.
- **Redirect bounding**: `MAX_REDIRECTS = 5`, each hop is revalidated through `validateFeedUrl` so a public server cannot 302 the bot to a private address (except via the two bypasses in findings #1 and #2, which this audit exists to fix).
- **Content-Type gating**: non-XML responses are rejected before they hit the parser — limits what a rogue server can feed into rss-parser.
- **HTTPS-only default**: `allow_http` defaults to `false`. Operators must explicitly opt in.
- **Permission gating**: every subcommand is behind a `+m` bind. NickServ verification is handled by the dispatcher per `require_acc_for` (no manual ACC calls in the plugin, as required).
- **Output sanitisation for IRC protocol**: `api.notice` and `api.say` both route through the core `sanitize()` at the plugin-api boundary, stripping `\r\n\0`. Finding #3 is about IRC _formatting_ codes, which is a different layer — this bullet is about IRC _protocol_ injection, which the plugin correctly delegates.
- **Config-defined feeds are protected from `!rss remove`**: `isRuntimeFeed` gates the delete path, so a `+m` operator cannot remove a feed the admin put in `plugins.json`.
- **Runtime feed corruption handling**: `loadRuntimeFeeds` catches JSON parse errors and deletes corrupt entries with a warning — no crash, no hang.
- **`stripHtmlTags` fixed-point loop**: correctly handles the nested-tag edge case that CodeQL flags on single-pass strip. The loop terminates because each non-terminal iteration strictly shortens the string.
- **Dedup hash**: SHA-1 of `guid || title+link` is deterministic and collision-resistant enough for a dedup table. No adversarial hash collision risk here — the worst case is a legitimate duplicate suppresses one announcement, not a security bypass.
- **Message queue integration**: `announceItems` goes through `api.say`, which goes through the core message queue — feed spam cannot flood the network.
- **Per-item drip delay**: 500ms between items in `announceItems` limits burst rate on top of the message queue.
- **Silent first-run seeding**: on initial load every current item is marked seen without announcing, preventing a feed replay on bot restart.

## Recommendations

1. **Fix the two SSRF bypasses first** (findings #1 and #2). Until both are done the URL validator gives a false sense of protection — any `+m` operator with basic DNS setup or IPv6 literacy can pivot the bot to localhost.

2. **Adopt `ipaddr.js`** for IP classification instead of hand-rolled regexes. The IPv6 check is too subtle to maintain by hand — every time a new bypass is found it'll be another round of regex patching. `ipaddr.js` has been audited, handles every notation class, and is already the de-facto standard in the Node SSRF-prevention libraries referenced below.

3. **Pin sockets to validated IPs.** The canonical Node fix for DNS rebinding SSRF is the `lookup` option passthrough on `http.get` — it's the same pattern used by `request-filtering-agent`, `ssrf-req-filter`, and `nossrf`. Use it here.

4. **Consider a shared SSRF helper** in `src/utils/` if any future plugin will make outbound HTTP requests (webhooks, image lookups, URL title fetchers). The validator + pinned-lookup pair is reusable and the risk of re-introducing a bypass in a second implementation is real. Extract once the second caller appears.

5. **Formalise the feed-content trust model**. Flag up in `plugins/rss/README.md` that feed content is attacker-controlled and that operators should only subscribe to feeds they trust. The plugin's job is to prevent network-level abuse (SSRF, DoS); it cannot prevent a hostile feed from saying misleading things in a channel. Setting operator expectations honestly is part of the security story.

6. **Add SSRF regression tests** covering: DNS rebinding (two-call lookup mock), hex-form IPv4-mapped IPv6, `0.0.0.0`, `169.254.169.254`, `127.0.0.1`, `10.x`, `172.16.x`, `192.168.x`, `::1`, `fe80::`, `fc00::`, `::ffff:127.0.0.1`, `::ffff:7f00:1`, userinfo URLs, and port `22`/`25` (once the port check lands). One test per bypass class, all under `tests/plugins/rss-url-validator.test.ts` (already exists — extend it).
