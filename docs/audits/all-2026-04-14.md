# Security Audit: all

**Date:** 2026-04-14
**Scope:** Every `.ts` file under `src/` and `plugins/`, plus `config/bot.example.json` and `config/plugins.example.json`. Baseline: `docs/SECURITY.md`.

## Summary

Hexbot's core security invariants — sanitize on inbound, parameterized SQL, STS, SASL PLAIN refusal, dispatcher verification gate, CTCP rate limit, message-queue flood cap, scrypt password hashing, frozen plugin API — are intact. The audit turned up one genuinely exploitable class (RSS SSRF / unbounded fetch), several defense-in-depth gaps that should be fixed before the next release, and a handful of info-level hardening recommendations.

**Findings:** **7 critical**, **28 warning**, **37 info**.

The three most urgent items:

1. **RSS plugin SSRF + unbounded body + billion-laughs** — the only trivially exploitable path by a compromised `+m` account, and the only one that reaches internal network resources.
2. **`.bot` audit metadata leaks plaintext command args** — future password-bearing subcommands will silently land in `mod_log`.
3. **auto-op binds with flag `'-'`**, bypassing the dispatcher's `VerificationProvider` gate — the classic NickServ-race CVE is closed today only by a correct-but-fragile manual check.

## Fix phases

Actionable items are grouped by phase below so downstream skills (`/build`, `/refactor`) can tick them off. Each box corresponds to one finding.

### Phase 1 — CRITICAL (fix before next deploy)

- [x] **RSS SSRF**: validate URLs in `plugins/rss/feed-fetcher.ts:37` and `plugins/rss/index.ts:219`. Require `https:` (or explicit http opt-in), DNS-resolve the host, and reject RFC1918 / loopback / link-local / ULA / `0.0.0.0/8` / `100.64.0.0/10`. Re-check after each redirect.
- [x] **RSS unbounded body**: pre-fetch via a custom `http.get` wrapper that aborts above a byte cap (e.g. 5 MB) and validates `Content-Type` before handing to `rss-parser`. (`plugins/rss/feed-fetcher.ts:37`)
- [x] **RSS billion-laughs**: reject any body containing `<!DOCTYPE` or pre-scan for entity declarations before xml2js parses. (`plugins/rss/feed-fetcher.ts:37`)
- [x] **`.bot` plaintext leak**: redact sensitive subcommands (`chpass`, etc.) from `reason`/`metadata.args` in `src/core/commands/botlink-commands.ts:494`. Maintain a denylist and refuse to relay them via `.bot` at all.
- [x] **`.chpass` transport gate**: in `src/core/commands/password-commands.ts:64`, replace `ctx.source === 'irc'` with `ctx.source !== 'repl' && ctx.source !== 'dcc'` so `botlink` CMD frames can never set passwords regardless of downstream ctx changes.
- [x] **auto-op bind flag**: in `plugins/chanmod/auto-op.ts:146`, either switch the bind to a flag in the op-trust set (so the dispatcher's `VerificationProvider` gate fires), or add a regression test asserting that every mode path in `grantMode` calls `verifyUser` when `require_acc_for` includes the target flag. Also consult `ctx.account` / the dispatcher account map for the fast path.
- [x] **Botlink password compare**: in `src/core/botlink/auth.ts:231`, replace `sent === this.expectedHash` with `crypto.timingSafeEqual(Buffer.from(sent), Buffer.from(this.expectedHash))` after a length check. (The existing TODO comment already acknowledges this.)

## Findings by area

### Dispatcher, IRC bridge, command handler, permissions, flood plugin

**No critical findings.** Core safety invariants (sanitize, SQL parameterization, ACC gate ordering, permission checks) all hold.

- **[WARNING]** `src/irc-bridge.ts:269-295` **ACTION bypasses flood check** — `onAction()` dispatches `pubm`/`msgm` without calling `dispatcher.floodCheck()`. Attacker can spam `/me` unthrottled. SECURITY.md §10.1 claim "once per IRC message" is violated. → call `floodCheck` with the same `ident@host` key used in `onPrivmsg`.
- **[WARNING]** `src/core/permissions.ts:298-314` **First-match wins in `findByHostmask`** — when two records have overlapping patterns, Map iteration order picks the winner. Violates SECURITY.md §3.3 ("must not short-circuit on the first matching hostmask if that hostmask belongs to a different user"). → score matches by specificity (fewest wildcards / longest literal prefix); or refuse overlapping adds at `addHostmask` time and log runtime multi-match.
- **[WARNING]** `plugins/flood/index.ts:100-108` **`isPrivileged` ignores account-tag** — `findByHostmask(hostmask)` called without account; users with only `$a:` patterns silently fail lookup and can be kicked by the flood plugin for normal activity. → resolve the nick's account and pass as second arg.
- **[WARNING]** `plugins/flood/index.ts:172-185` **`handleNickFlood` per-channel re-resolution** — privilege check re-runs per channel inside the iteration. Nick changes are global; resolve privilege once up front.
- **[WARNING]** `src/irc-bridge.ts:435-438` **Account tag trust without cap gate** — `extractAccountTag` used even if `account-tag` cap isn't negotiated. Defense in depth. → gate on `capabilities.hasCap('account-tag')`.
- **[WARNING]** `src/command-handler.ts:169-179` **Command handler drops inbound `ctx.account`** — synthesized `HandlerContext` omits `account`, so `$a:` fast path can miss in the window before `channel-state` records the account. → thread `ctx.account` from the PRIVMSG flow into the command ctx.
- **[INFO]** `src/dispatcher.ts:236-246` **Time bind flags are meaningless** — timer binds don't run through `dispatch()`, so declared flags are ignored. → reject non-`-` flags on `time` binds at bind time.
- **[INFO]** `src/core/permissions.ts:460-471` **Dead `d` flag** — `VALID_FLAGS = 'nmovd'` but `d` has no semantics anywhere.
- **[INFO]** `plugins/flood/lockdown.ts:50-53` **`flooders` set never pruned** — Set grows while `timestamps` array is pruned; eventually the hostmask is permanently suppressed from triggering lockdown (fails open to the flooder).
- **[INFO]** `src/core/permissions.ts:409-422` `saveToDb` rewrites every user on every mutation — amplification of DB writes on permission edits. Not exploitable.
- **[INFO]** `plugins/flood/enforcement-executor.ts:127-140` `buildFloodBanMask` — bridge already sanitizes hostname, but calling `sanitize` on the built `banMask` would be defense in depth.
- **[INFO]** `src/irc-bridge.ts:595-615` unknown-command 477 path — `params[1]` sanitize ordering verified safe.

**Passed:** inbound `sanitize()` on every field of every event; dispatcher verification gate runs before handler call; fail-closed on no matching record; `pub`+`pubm` and `msg`+`msgm` share one flood counter; owner-flag flood bypass uses full permission lookup; case-insensitive handle/channel compares; insecure-hostmask warnings fire for `nick!*@*` on privileged users; `$a:` patterns skip the warning; handler try/catch containment; `unbindAll` clears timers; CTCP rate-limit keyed on `ident@host`; REPL/botlink source bypasses the IRC flag gate by design.

An attacker who wins the CTCP race can stream gigabytes without a newline, buffered in-process. 30s prompt timeout doesn't cover fast bulk delivery. → cap line length explicitly (≈4 KiB) and destroy sockets that exceed it.

- **[INFO]** Blank prompt lines reset the idle timer indefinitely; cap at 3 blank re-prompts.
- **[INFO]** DCC authorization uses live hostmask every command; vhost-persistent networks rely entirely on the password to distinguish identities (documented).
- **[INFO]** Lost CTCP race is documented; optionally notify the legitimate user.
- **[INFO]** `setKeepAlive(60_000)` is fine; noted for completeness.

### Botlink subsystem

- **[CRITICAL]** `src/core/botlink/auth.ts:231-234` **`verifyPassword` is not timing-safe** — `sent === this.expectedHash` with an existing TODO. Combined with the fixed-salt scrypt, the wire hash is the only credential needed for replay; a timing oracle lets an attacker iteratively learn it. → `crypto.timingSafeEqual(Buffer.from(sent), Buffer.from(this.expectedHash))` after length check.
- **[WARNING]** `src/core/botlink/protocol.ts:46-51` + `leaf.ts:267-273` **Replay-able hash credential** — fixed salt + no challenge-response means a packet capture on the plaintext link re-authenticates from any non-banned IP. → add hub `HELLO_CHALLENGE` + `HMAC(nonce, password)`; document the tunnel-over-WireGuard requirement loudly.
- **[WARNING]** `src/core/botlink/relay-router.ts` **Pending route maps have no hard cap** — slow attacker CMD/PROTECT traffic plants entries; only TTL sweeps reclaim. → `MAX_PENDING_ROUTES` guard with overflow rejection.
- **[WARNING]** `src/core/botlink/protect.ts:104-113` **`PROTECT_UNBAN`/`PROTECT_INVITE` skip recognition check** — UNBAN issues `mode(channel, '-b', nick)` for any string; a leaf can send `nick: '*!*@*'` and unban everyone. INVITE invites arbitrary nicks. Other PROTECT\_\* request frames correctly check `isRecognized()`. → add recognition check (and for UNBAN, require a specific non-catch-all mask).
- **[WARNING]** `src/core/botlink/protocol.ts:26-40` + `hub.ts:633-635` **PROTECT\_\* requests fan out leaf-to-leaf** — they are NOT in `HUB_ONLY_FRAMES`, so any leaf can ask any other leaf to act. - **[WARNING]** `src/core/botlink/relay-handler.ts:118-150` **Relay session doesn't re-verify handle exists on every frame** — `RELAY_INPUT` / party-chat keep running after `permissions.getUser(handle)` returns null. → send `RELAY_END` and drop session.
- **[INFO]** CIDR scan on every connection runs before pending increment (correct order); unconditional 500-entry linear scan is fine at current cap.
- **[INFO]** `sanitizeFrame` recursion has no depth limit; nested JSON up to 64 KB could stack-overflow.
- **[INFO]** `CHAN_BAN_ADD enforce:true` with broad mask can affect many nicks; cap affected nicks.
- **[INFO]** `frame.type` not length/charset-restricted; `startsWith('PROTECT_')` test on a 60k-char string wastes cycles.
- **[INFO]** BSAY fan-out doesn't re-verify caller's `+m` on the hub side; any authenticated leaf can spam hub channels.
- **[INFO]** `isWhitelisted` is IPv4-only; IPv6 clients silently bypass whitelist.
- **[INFO]** Outbound frames don't pass through `sanitizeFrame` (inbound only).

## Recommendations

1. **Ship Phase 1 before the next deploy.** The RSS plugin is not safe on a network where any `+m` user may be compromised — SSRF + unbounded body are trivially exploitable primitives. Everything else in Phase 1 is defense-in-depth but cheap.
2. **Add a regression test** that asserts non-chanmod plugins cannot see `nick_recovery_password`. The `PluginBotConfig` strip is the documented seal; this audit is the second time a shallow-copy has broken it.
3. **Consolidate permission-mutation audit logging.** `Permissions.recordModAction` hardcoding `source: 'system'` is silently destroying a forensic dimension the audit pipeline is designed to preserve. Plumbing `ctx.source` through is a small refactor with outsized value.
4. **Refactor auto-op and invite to bind with flags.** The dispatcher's verification gate exists precisely so plugin authors don't have to get ACC verification right. Bypassing it with `'-'` flags is the architectural anti-pattern SECURITY.md §3.2 warns against.
5. **Revisit the `verifyPassword` TODO** — it is already marked "security audit WARNING" in the source and has been open long enough to appear in a follow-up audit.
6. **Consider a challenge-response handshake** for bot-link. Even with mandatory VPN, stripping the replay primitive closes the failure mode where one machine in a trusted network is compromised and passively sniffed.

No critical findings in core dispatcher / IRC bridge / services / DCC / utilities. The chanmod critical is latent (closed by a correct manual check today). The command-layer criticals are about future-proofing audit log hygiene and transport gating. The RSS criticals are the only ones that are exploitable today.
