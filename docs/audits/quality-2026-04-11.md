# Quality Report — hexbot

**Date:** 2026-04-11
**Scanned:** 77 `.ts` files across `src/` and `plugins/`

## Summary

Overall health is good — the core plumbing (dispatcher, irc-bridge, command-handler, database, config, event-bus, repl, channel-state, permissions, services, message-queue) is clean, single-purpose, and navigable. The codebase's structural debt is concentrated in exactly four places: `botlink-hub.ts`, `dcc.ts`, `mode-enforce.ts`, and `botlink-commands.ts`. Every one of these grew organically over feature additions and now bundles enough distinct responsibilities that a cold reader has to page through hundreds of lines to find the piece they care about. A handful of systemic duplications (arg parsing, hostmask construction, hub/leaf branching, "is this the bot's nick?") would benefit from a single coordinated cleanup rather than file-by-file fixes. The trust-boundary files (irc-bridge, sanitize, botlink-protocol, plugin-loader API factory) are worth emulating.

## High Priority

### `src/core/botlink-hub.ts` (1042 lines) — god class mixing auth + connection + relay + party line + rate limiting

**Problem:** `BotLinkHub` owns IP ban tracking, handshake authentication, per-connection rate limiters, per-leaf heartbeat, frame routing, party line state, relay sessions, protection routing, and command-result routing. You cannot describe it in one sentence without at least three "and"s. Every piece of auth math (exponential ban backoff, sliding-window brute-force detection, whitelist checks) is glued to frame routing, so unit-testing the escalation logic requires a full `BotLinkHub` instance.

**Evidence:**

- Class body spans lines 123–1042. Eight distinct `Map` fields (`remotePartyUsers`, `activeRelays`, `cmdRoutes`, `protectRequests`, `authTracker`, `pendingHandshakes`, `manualCidrBans`, `leaves`).
- `recordAuthFailure()` + `sweepStaleTrackers()` auth logic (lines 815–874) is ~60 lines intermixed with routing methods.
- `onSteadyState()` (lines 880–977, ~97 lines) chains: identity rewrite → rate limit → fan-out → frame-type switch → dispatch to 8+ handlers.
- Relay state tracking (lines 126–133) uses four parallel maps with different semantics but no shared abstraction.

**Suggested split:**

1. `src/core/botlink-auth.ts` — a `BotLinkAuthManager` class owning `authTracker`, `pendingHandshakes`, `manualCidrBans`, and the exponential ban math. Unit-testable without TCP.
2. `src/core/botlink-relay-state.ts` — a `RelaySessionTracker` owning `remotePartyUsers`, `activeRelays`, `cmdRoutes`, `protectRequests` and the `cleanupLeafState()` path.
3. Keep `BotLinkHub` as the TCP-facing orchestrator: handshake, fan-out, delegation to the above.
4. Extract `enforceLeafRateLimit(frame, conn)` out of `onSteadyState()`.

**Risk:** Medium. Auth is security-sensitive; the extraction must be behavior-preserving and covered by the existing auth tests before merge.

### `plugins/chanmod/mode-enforce.ts` (~920 lines) — single file owns 11 distinct enforcement concerns

**Problem:** A single `setupModeEnforce()` wires 11 handler functions, plus three top-level helpers (`performMassReop`, `performHostileResponse`, `punishDeop`), plus `syncChannelModes()`, plus the main `mode` bind that chains eight of them in sequence. A reader trying to answer "what happens when the bot is deopped?" has to trace through the whole file because threat-level state, cooldown state, and recovery actions are all intermixed.

**Evidence:**

- Handlers: `handleThreatModeLockdown` (54–70), `handleReapplyRemovedModes` (73–92), `handleRemoveUnauthorizedModes` (100–120), `handleChannelKeyEnforcement` (123–155), `handleChannelLimitEnforcement` (158–193), `handleBotSelfDeop` (199–262), `handleBotOpped` (268–321), `handleBitchMode` (327–363), `handleBotBanned` (366–392), `handleEnforceBans` (398–419), `handleUserModeEnforcement` (422–511).
- `performMassReop` (722–790) and `performHostileResponse` (802–880) are conceptually "recovery actions" but live at the file tail.
- Main `mode` bind (620–689) reads as an 8-call chain whose ordering is load-bearing but not documented.

**Suggested split:**

- `mode-enforce-channel.ts` — channel-mode enforcement: reapply-removed, remove-unauthorized, key, limit, `syncChannelModes` (~250 lines).
- `mode-enforce-user.ts` — user-mode enforcement: bitch mode, user mode enforcement, punish-deop (~300 lines).
- `mode-enforce-recovery.ts` — bot-deopped handling, bot-opped handling, mass re-op, hostile response (~250 lines).
- `mode-enforce.ts` remains as the thin wiring module: threat lockdown, enforcebans, `syncChannelModes` re-export, and the main bind that orders the handlers (~150 lines).

**Risk:** Medium. The handlers share state via `SharedState` and threat callbacks; tests covering mode enforcement + recovery are the safety net. Test first, then split.

### `src/core/dcc.ts` (814 lines) — `DCCSession` + `DCCManager` + port allocation + 67-line guard chain

**Problem:** Two large classes plus a port allocator plus the DCC subcommand router plus a 67-line `rejectIfInvalid()` guard chain, all in one file. The guard chain sequentially runs passive-DCC parsing, hostmask validation, flag check, session-limit check, and NickServ verification — any one of which should be independently testable, but none are.

**Evidence:**

- `DCCSession` (217–451, ~235 lines) owns session lifecycle, relay mode, idle timeout, and command dispatch.
- `DCCManager` (460–814, ~355 lines) owns TCP server creation, port allocation, session tracking, CTCP parsing, and the guard chain.
- `rejectIfInvalid()` (634–700) is a 67-line top-to-bottom if-cascade.
- `openDccServer()` (729–778) mixes TCP server creation, timeout setup, and connection handling — all `v8 ignore`d because it's untestable without a real TCP socket.

**Suggested split:**

1. Extract `DCCGuards` with one method per check (`checkPassive`, `checkHostmask`, `checkFlags`, `checkSessionLimit`, `checkNickServVerify`). Each becomes independently mockable.
2. Move `DCCSession` to `src/core/dcc-session.ts`.
3. Keep `DCCManager` as the TCP + orchestration surface; its `onDccCtcp` becomes a short "run guards, then spawn server, then create session" method.

**Risk:** Medium. DCC has race-condition mitigations documented in `SECURITY.md` §3.4 — the existing DCC tests need to stay green throughout.

### `src/core/commands/botlink-commands.ts` (527 lines) — 139-line `.botlink` switch + hub/leaf branching repeated 6+ times

**Problem:** `registerBotlinkCommands` contains a 139-line `.botlink` sub-command router (lines 33–171) where each case re-implements the same three guards (`config?.enabled`, `hub vs leaf`, argument validation) with minor variations. The same `if (hub) { ... } else if (leaf) { ... } else { ctx.reply('Not connected') }` pattern appears 6+ times in the file with no abstraction.

**Evidence:**

- `.botlink` sub-command cases: status (42–67), bans (122–150), ban (152–170), and others — each a ~20-line block with its own nesting.
- `.whom` (334–368), `.bot` (378–429), `.bsay` (460–500) all repeat the hub/leaf detection.
- `.bot` (378–429) and `.bsay` (460–500) both build near-identical `RELAY_REQUEST` frames with the `fromBot` / `handle` fields.
- `.ban` inside `.botlink` duplicates the duration-parsing pattern from `src/core/commands/ban-commands.ts` lines 111–116.

**Suggested split:**

1. Extract each `.botlink` sub-command to its own named handler (`handleBotlinkStatus`, `handleBotlinkBan`, …) and dispatch via a map.
2. Extract `withBotLink(ctx, fn)` helper: checks `config?.enabled`, resolves hub-or-leaf, and calls `fn(link)`. Eliminates all 6+ branches.
3. Extract `buildRelayRequestFrame(handle, targetBot, command)` to stop the duplication between `.bot` and `.bsay`.

**Risk:** Low — this is a pure refactor; the bind flags and permission checks stay exactly as they are.

## Medium Priority

### `src/plugin-loader.ts` (1051 lines) — loader lifecycle + API factory in the same file

**Problem:** The file has two separate jobs: plugin lifecycle (discover/load/unload/reload, lines 110–399) and API construction (`createPluginApi` + 7 sub-factories, lines 405–1051). The API factories are ~420 lines of wrapping code — 40% of the file — and are orthogonal to load lifecycle.

**Evidence:** `createPluginApi()` at 458–522 is 64 lines of object-literal assembly. Sub-factories: `createPluginIrcActionsApi`, `createPluginChannelStateApi`, `createPluginPermissionsApi`, `createPluginServicesApi`, `createPluginDbApi`, `createPluginBanStoreApi`, `createPluginChannelSettingsApi`, `createPluginHelpApi`, `createPluginLogApi` — each 20–60 lines.

**Suggested split:** Extract `src/plugin-api-factory.ts` with all the sub-factories and the top-level `createPluginApi`. Loader imports and calls it. No behavior change; clear testability boundary between "what loads plugins" and "what shape of API plugins see."

**Risk:** Medium. The factories currently capture loader internals (`this.db`, `this.botConfig`, `this.channelState`, etc.). The refactor means threading 15–20 dependencies through a single `createPluginApi(deps)` call — mechanical but touches many call sites.

Not a god file — every piece is genuinely orchestration — but it's past the point where a reader can hold the wiring in their head.

**Evidence:** `start()` at 225–412. Constructor at 113–222. Private methods `_relayDeps`, `handleProtectFrame`, `handleIncomingBotlinkFrame`, `wirePartyLine` (650–740) are botlink-wiring glue.

**Suggested split:** Extract `src/botlink-wiring.ts` containing the `_relay*` helpers, `handleIncomingBotlinkFrame`, and `wirePartyLine`. `bot.ts` calls `wireBotlink(this, role)` during `start()`. Drops ~100 lines from bot.ts and turns the wiring into a named, testable concern.

**Risk:** Low. The extracted code is stateless closures over `this` — threading them as dependencies is mechanical.

## Patterns to address across the codebase

These appear in multiple files — fix them once, in a dedicated refactor pass, rather than file-by-file.

1. **Ad-hoc `${ctx.nick}!${ctx.ident}@${ctx.hostname}` construction.** Appears in `greeter` (3×), `flood`, `chanmod/invite`, `chanmod/auto-op`, `chanmod/commands`, and several command handlers. Add `buildHostmask(ctx)` to `src/utils/` or (better) expose it as `api.hostmask` / `ctx.hostmask` so plugins don't construct it at all. Behavior-preserving across all call sites.

2. **"Is this the bot's own nick?" check.** `isBotNick(api, nick)` exists in `plugins/chanmod/helpers.ts`; `flood/index.ts` reinvents it; `topic/index.ts` checks `ctx.nick === ...` implicitly. Move the helper to a shared location (either the plugin API or a core util) so every plugin has one canonical path.

3. **Ad-hoc permission flag checks.** `flood/isPrivileged` and `greeter/meetsMinFlag` each implement their own flag-hierarchy logic. Both should call `api.permissions.checkFlags(requiredFlags, ctx)`. This centralizes the `n` / `m` / `o` / `v` / `d` semantics and guarantees the `n`-implies-all rule can't drift.

4. **Command-handler arg-parsing duplication.** Every `src/core/commands/*.ts` file contains `args.trim().split(/\s+/)` followed by index access and ad-hoc validation. A small helper — `parseArgs(args, { min, max, schema })` or `splitN(args, n)` — would centralize the shape and catch the inconsistencies (e.g., `.say` requires a `#`-prefix target while `.msg` does not). At minimum, define a shared `parseTargetMessage(args)` for the `.say` / `.msg` / `.bsay` / `.notice` family.

5. **Duration-parsing repetition.** `src/utils/duration.ts` exists and is used by `.ban`. `.botlink ban` (inside `botlink-commands.ts`) reimplements the same parser inline. Route everything through `parseDuration`.

6. **Hub-vs-leaf branching in `botlink-commands.ts`.** Six call sites use the same `if (hub) ... else if (leaf) ... else ctx.reply('Not connected')` pattern. A `withBotLink(bot, ctx, (link) => ...)` helper collapses all of them.

7. **`ctx.source` checks across the three command sources.** `permission-commands.ts`, `botlink-commands.ts`, and `irc-commands-admin.ts` each reason about `ctx.source === 'repl'`, `!== 'dcc'`, `!== 'botlink'` with different rules. Introduce named predicates (`isTrustedSource(ctx)`, `sourceLabel(ctx)`) so the intent is explicit and consistent.

8. **`nick!*@*` insecure-hostmask awareness.** `permissions.ts` warns on insecure admin patterns. Command handlers that accept hostmasks (`adduser`, `addhostmask`) could share a `validateHostmaskPattern()` helper so the UX is uniform across the CLI.

## What looks good

Call these out deliberately — they are worth emulating, and they should not be touched in a cleanup pass.

- **`src/dispatcher.ts` (445 lines)** — single responsibility, well-factored flood-limit + verification-gate + flag-check pipeline. The order of operations in `dispatch()` is the right shape for a security-critical hot path.
- **`src/irc-bridge.ts` (562 lines)** — consistent "sanitize → build context → dispatch" shape across every event handler. This is the trust boundary and it is uniformly applied.
- **`src/command-handler.ts` (213 lines)** — transport-agnostic routing that works for REPL + IRC + DCC + botlink without any per-transport special cases. Minimal and correct.
- **`src/database.ts`, `src/config.ts`, `src/logger.ts`, `src/repl.ts`, `src/event-bus.ts`, `src/index.ts`** — each file is ≤400 lines, each owns exactly one concern, none manufactures abstractions.
- **`src/config.ts` two-stage resolution** — `parseBotConfigOnDisk` → `resolveSecrets` → `validateResolvedSecrets` is cleanly pipelined, and the Zod schema has a compile-time guard against drift from `BotConfigOnDisk`.
- **`src/core/botlink-protocol.ts`** — frame I/O + rate counter + scrypt hashing in 283 lines. `sanitizeFrame` is recursive and correct; `executeCmdFrame` is the right extraction to stop hub/leaf duplication.
- **`src/core/botlink-sharing.ts`, `botlink-sync.ts`, `botlink-relay-handler.ts`, `botlink-protect.ts`** — each owns exactly one botlink concern. These are the model for how `botlink-hub.ts` should look after the refactor above.
- **`src/core/channel-state.ts`** — the canonical place for per-channel tracking. The fact that no plugin reinvents channel tracking is because this file exists and is good.
- **`src/core/channel-settings.ts`, `ban-store.ts`, `message-queue.ts`, `help-registry.ts`, `irc-commands.ts`, `connection-lifecycle.ts`, `memo.ts`, `services.ts`, `permissions.ts`** — every one of these is ≤400 lines, focused, and clean.
- **`src/core/commands/dispatcher-commands.ts` (45 lines)** — the model for how a command module should look: tiny, one command, one responsibility.
- **`src/utils/*.ts`** — uniformly small, single-purpose, correct. `wildcard.ts` wildcard DP is worth reading as a reference. `sanitize.ts` at 5 lines is the right size for what it does.
- **`plugins/chanmod/index.ts`, `state.ts`, `helpers.ts`, `auto-op.ts`, `protection-backend.ts`, `takeover-detect.ts`, `join-recovery.ts`, `topic-recovery.ts`, `anope-backend.ts`, `atheme-backend.ts`, `bans.ts`, `invite.ts`, `sticky.ts`** — despite chanmod's size, 13 of its 17 files are clean and well-factored. The problems are concentrated in `mode-enforce.ts`, `protection.ts`, `commands.ts`, and `chanserv-notice.ts` — everything else is fine.
- **`plugins/chanmod/protection-backend.ts` + `anope-backend.ts` + `atheme-backend.ts`** — the `ProtectionChain` + `ProtectionBackend` abstraction is a legitimate backend dispatcher, not over-engineering. Each backend impl is 200–280 lines of service-specific logic with no cross-contamination.
- **`plugins/chanmod/takeover-detect.ts`** — rolling threat scoring + escalation is clean, correct, and unit-testable in isolation.
- **`plugins/chanmod/join-recovery.ts`** — exponential backoff + sustained-presence reset guards against attacker-triggered ChanServ spam. Good defensive design.
- **`plugins/seen/index.ts`, `plugins/help/index.ts`, `plugins/8ball/index.ts`** — small focused plugins with proper init/teardown hygiene. `help/index.ts` in particular handles permission filtering and category grouping in under 150 lines.
- **`plugins/chanmod/state.ts` factoring** — the shared-state object groups its fields by feature concern (mode enforcement / cycle / stopnethack / takeover / kick recovery / topic recovery) with comments, making a potentially grab-bag structure navigable.
